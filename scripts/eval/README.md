# Eval — оценка качества поиска по вики

`scripts/eval.ts` — это отдельная CLI-утилита, которая измеряет **качество извлечения**
(retrieval) вики-пайплайна: насколько хорошо запрос находит нужные страницы.
Она прогоняет фиксированный набор пар «вопрос → эталонные страницы» (gold-набор)
через тот же код отбора seed-страниц и BFS-расширения, что и обычный запрос
(`src/phases/query.ts`), и считает метрики **Recall@k** и **MRR**.

Цель — сравнивать конфигурации (например `dense` против `jaccard`, разную глубину
BFS) и видеть, стало ли лучше или хуже: до изменения и после.

---

## Что измеряется

Для каждого вопроса утилита получает ранжированный список найденных страниц и
сравнивает его с эталонными id из gold-набора.

- **Recall@k** — доля эталонных страниц, попавших в топ-`k` результатов
  (k = 3, 5, 8). `1.0` = все эталонные страницы найдены в пределах топ-k.
- **MRR** (Mean Reciprocal Rank) — средний обратный ранг первого попадания.
  Если нужная страница на 1-м месте → `1.0`, на 2-м → `0.5`, не найдена → `0`.

Метрики считаются по двум слоям:

- **seed** (`s` в таблице) — только страницы-кандидаты, отобранные на первом шаге.
- **union** (`u` в таблице) — seed плюс страницы, добавленные обходом графа (BFS).

И по двум режимам отбора seed-страниц:

- **jaccard** — пересечение токенов вопроса и аннотаций. Работает **без ключа и без
  сети** (keyless).
- **dense** — семантический поиск по эмбеддингам. Требует endpoint эмбеддингов
  и заполненный кэш `_config/_embeddings.json` (см. «Режим dense» ниже).

## Предварительные требования

1. Установить зависимости (ставит `tsx`, на котором запускается утилита):

   ```bash
   npm install
   ```

2. Хранилище Obsidian с вики нужной структуры:

   ```
   <vault>/!Wiki/<подпапка>/_config/_index.md   ← обязательно
   <vault>/!Wiki/<подпапка>/.../*.md            ← страницы (могут быть во вложенных папках)
   ```

   Если под `!Wiki/` несколько подпапок — укажите нужную флагом `--wiki <подпапка>`.
   Если подпапка одна — она определяется автоматически.

## Быстрый старт (jaccard, без ключа)

```bash
npm run eval -- \
  --vault /путь/к/vault \
  --gold scripts/eval/example.gold.json \
  --config jaccard \
  --wiki os
```

`example.gold.json` — это **шаблон** с заглушечными id (`Ingest`, `Embedding-Cache`,
`Query-Graph-Traversal`). Их почти наверняка нет в вашем хранилище, поэтому утилита
выведет предупреждения «not present in vault» и все метрики будут `0.000`. Это
нормально — нужно создать свой gold-набор (см. ниже).

---

## Как сформировать gold-набор

Gold-набор — это JSON-массив пар `{ "q", "gold" }`, специфичный для вашего хранилища
(рекомендуется 30–50 вопросов для устойчивых чисел):

```json
[
  { "q": "настройка прокси в macOS",          "gold": ["wiki_os_macos"] },
  { "q": "UFW брандмауэр firewall Ubuntu",     "gold": ["wiki_os_ufw"] },
  { "q": "NFS сервер экспорт каталогов",       "gold": ["wiki_os_nfs_kernel_server", "wiki_os_exportfs"] }
]
```

- `q` — вопрос пользователя (на любом языке, как реально спрашивают).
- `gold` — список **pageId-стемов**: это имя файла без расширения и без папки.
  Например, для файла `!Wiki/os/systems/wiki_os_macos.md` стем — `wiki_os_macos`.
  Одна тема может покрываться несколькими страницами → в `gold` можно указать
  больше одного id.

**Где взять стемы.** Откройте индекс вики `!Wiki/<подпапка>/_config/_index.md`.
Строки выглядят так:

```
- [[wiki_os_macos]] systems/wiki_os_macos.md — Операционная система macOS ...
```

Стем внутри `[[...]]` (`wiki_os_macos`) и есть нужный pageId.

Сохраните свой набор рядом с шаблоном, например
`scripts/eval/<имя-вашего-vault>.gold.json`. Gold-файлы содержат только вопросы и id
(никакого контента хранилища), поэтому их **безопасно коммитить**.

## Пошаговая инструкция

1. **Установите зависимости:** `npm install`.
2. **Выберите хранилище** со структурой `<vault>/!Wiki/<подпапка>/_config/_index.md`
   и запомните имя подпапки (например `os`).
3. **Соберите id страниц:** откройте `_config/_index.md`, выпишите стемы из `[[...]]`,
   которые должны находиться по вашим вопросам.
4. **Создайте gold-набор** `scripts/eval/<vault>.gold.json` в формате выше.
5. **Снимите базовую точку** (baseline) и сохраните её в файл:

   ```bash
   npm run eval -- --vault /путь/к/vault --gold scripts/eval/<vault>.gold.json \
     --config jaccard --wiki os --out /tmp/baseline.json
   ```

6. **Прочитайте таблицу** (см. «Как читать вывод»).
7. **Измените конфигурацию** (например глубину BFS) и **сравните с baseline** —
   рядом с каждой ячейкой появятся дельты:

   ```bash
   npm run eval -- --vault /путь/к/vault --gold scripts/eval/<vault>.gold.json \
     --config jaccard --wiki os --bfs-depth 2 --baseline /tmp/baseline.json
   ```

8. **Сравните режимы** dense и jaccard в одном прогоне (нужен endpoint, см. ниже):

   ```bash
   EVAL_EMBED_BASE_URL=http://localhost:11434 \
     npm run eval -- --vault /путь/к/vault --gold scripts/eval/<vault>.gold.json \
     --config dense,jaccard --wiki os
   ```

## Как читать вывод

```
config                 sR@3     sR@5     sR@8     sMRR     uR@3     uR@5     uR@8     uMRR
jaccard               1.000    1.000    1.000    1.000    1.000    1.000    1.000    1.000
```

- Каждая строка — одна конфигурация (`--config`).
- `s*` — метрики по слою **seed**, `u*` — по слою **union** (seed + BFS).
- `R@3 / R@5 / R@8` — Recall@k, `MRR` — средний обратный ранг. Все значения в `[0, 1]`.

С флагом `--baseline` рядом с каждой ячейкой выводится дельта относительно
сохранённого прогона:

```
jaccard               0.800 (▲+0.100)   ...   0.500 (=)   ...
```

- `▲+0.100` — стало лучше на 0.100;
- `▼-0.050` — стало хуже;
- `(=)` — без изменений.

## Все флаги CLI

| Флаг | По умолчанию | Назначение |
|------|--------------|-----------|
| `--vault <path>` | — (обязательный) | Путь к хранилищу Obsidian. |
| `--gold <gold.json>` | — (обязательный) | Путь к gold-набору. |
| `--wiki <подпапка>` | автоопределение | Подпапка вики под `!Wiki/`. Нужна, если подпапок несколько. |
| `--config dense\|jaccard` | `dense,jaccard` | Список конфигураций через запятую. |
| `--bfs-depth N` | `1` | Глубина BFS-расширения для слоя union. |
| `--top-k N` | `8` | Сколько seed-кандидатов отбирать. |
| `--out run.json` | — | Сохранить снимок метрик в файл (для сравнения позже). |
| `--baseline run.json` | — | Сравнить текущий прогон с ранее сохранённым (дельты). |

## Режим dense (эмбеддинги)

`dense` использует семантический поиск. Параметры берутся так:

- `model` и `dimensions` — из заголовка кэша `!Wiki/<подпапка>/_config/_embeddings.json`
  (кэш должен быть заранее построен плагином);
- `baseUrl` — из переменной окружения `EVAL_EMBED_BASE_URL`;
- `apiKey` — из `EVAL_EMBED_API_KEY` (**необязательно**: keyless-Ollama поддерживается).

```bash
EVAL_EMBED_BASE_URL=http://localhost:11434 \
EVAL_EMBED_API_KEY=                                  \
  npm run eval -- --vault /путь/к/vault --gold scripts/eval/<vault>.gold.json \
  --config dense,jaccard --wiki os
```

Если endpoint не задан (`EVAL_EMBED_BASE_URL` пуст), `dense` **не падает**, а выводит
предупреждение и внутренне откатывается к jaccard — прогон не помечается ложно как
«dense».

## Частые ситуации

- **`gold id "X" not present in vault — counts as a miss`** — указанного id нет среди
  страниц хранилища (опечатка в стеме или устаревший gold). Засчитывается как промах.
- **`multiple wiki subfolders found ...`** — под `!Wiki/` несколько подпапок; добавьте
  `--wiki <подпапка>`.
- **`no wiki subfolder with _config/_index.md found ...`** — в хранилище нет вики нужной
  структуры; проверьте путь `--vault` и наличие `_config/_index.md`.
- **Все метрики `0.000`** — обычно из-за заглушечных id шаблона; создайте свой
  gold-набор с реальными стемами из `_index.md`.
