# Конвенции написания wiki-страниц

## Язык и стиль

### Основные правила
- **Основной язык**: русский
- **Технические термины** (SQL, API, LLM, ETL, SCD, TTL, DDL, JSON, YAML) — не переводить
- **Имена систем**: оригинальное написание — RT.DataExporter, CRM B2C, ЦХД, СИП
- **Аббревиатуры**: расшифровывать при первом использовании на странице
  - Пример: "ЦХД (Центральное хранилище данных)"
- **Стиль**: нейтральный, информативный, без разговорных оборотов и оценочных суждений

### Недопустимые формулировки
- "Очевидно, что..." / "Понятно, что..."
- "Лучший способ..." (без атрибуции источника)
- "Кажется, что..." / "По всей видимости..."
- Местоимения "я", "мы", "наш"

## Именование файлов и папок

### Файлы wiki-страниц
- Формат: kebab-case, кириллица допустима
- Примеры корректных имён:
  - `версионирование-scd.md`
  - `rt-data-exporter.md`
  - `clickhouse-обзор.md`
  - `поток-gp-s3-ch.md`
- Запрещено: пробелы, точки (кроме расширения), спецсимволы кроме дефиса

### Папки доменов
- Нижний регистр, латиница (для надёжности WikiLink на case-sensitive FS)
- Подпапки аналогично: `архитектура-данных/`, `системная-архитектура/`

### Заголовки страниц (H1)
- Русское название концепции или системы
- Технический термин в скобках при необходимости: `# Версионирование (SCD1/SCD2)`
- Для имён систем: `# RT.DataExporter`

## Структура wiki-страницы

### Обязательные элементы (в порядке следования)

**1. Frontmatter (YAML)**
```yaml
---
wiki_sources: [пути к источникам]
wiki_updated: YYYY-MM-DD
wiki_status: stub | developing | mature
tags: [теги]
aliases: [альтернативные названия]
---
```

**2. Заголовок H1**
```markdown
# Название сущности
```

**3. Вводный абзац (определение)**
Сразу после H1, без заголовка раздела. 1-3 предложения — что это такое, зачем нужно.

**4. Раздел "Основные характеристики" (H2)**
```markdown
## Основные характеристики
```
Ключевые свойства и параметры сущности.

**5. Раздел "Связанные концепции" (H2)**
```markdown
## Связанные концепции

- [[другая-страница-вики]]
- [[ещё-одна-страница|отображаемое имя]]
```

### Опциональные разделы (добавлять при наличии информации)

- `## Применение в контексте [Домен]` — специфика применения
- `## Примеры` — конкретные примеры из источников
- `## Ограничения` — известные ограничения
- `## Best Practices` — рекомендации из источников
- `## История изменений` — если сущность эволюционировала (версии, даты)

## WikiLinks

### Когда использовать
- Ссылаться на другие wiki-страницы через `[[имя-страницы]]`
- При несовпадении имени файла с нужным текстом: `[[имя-файла|отображаемый текст]]`

### Запрещено
- Создавать "мёртвые" ссылки — только на уже существующие страницы вики
  - Исключение: можно создать ссылку с пометкой `<!-- TODO: создать страницу -->` если сущность явно важна
- Ссылаться на источники через WikiLinks — только через `wiki_sources` в frontmatter
- Ссылаться на файлы вне `.wiki/` через WikiLinks (только через `wiki_sources`)

## Правила работы с контентом

### Синтез, не копирование
- Информацию из источников переработать и структурировать
- Дословные цитаты допустимы только в code-блоках (SQL, конфигурации)

### Frontmatter — wiki_status
| Значение | Критерии |
|----------|---------|
| `stub` | Страница создана, < 2 источников, < 10 предложений контента |
| `developing` | ≥ 2 источника, ≥ 10 предложений, основные разделы заполнены |
| `mature` | ≥ 4 источника, все ключевые разделы, стабильный контент |

## Запрещено на wiki-страницах

1. **Placeholder-текст**: "TODO", "см. источник", "будет добавлено"
2. **Мнения без атрибуции**: "лучший подход — ..." (без указания источника)
3. **Дублирование страниц**: перед созданием проверять `index.md`
4. **Удаление существующей информации**: только добавление и обновление
5. **Пустые разделы**: если раздел нечем заполнить — не создавать его
