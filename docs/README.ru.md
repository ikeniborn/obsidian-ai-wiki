# AI Wiki — плагин Obsidian

[English version →](../README.md)

Автоматически строит и пополняет базу знаний из ваших заметок с помощью ИИ-ассистента.

**Почему AI Wiki:**
- **Офлайн по умолчанию** — работает с Ollama или любым локальным ИИ-сервером; ваши заметки не уходят с вашего компьютера
- **Растёт вместе с заметками** — каждый Ingest добавляет новые темы и автоматически обновляет существующие
- **Прозрачность** — каждый шаг ИИ виден в реальном времени в боковой панели
- **Два ИИ-бэкенда** — локальный (Ollama / OpenAI-совместимый) или Claude; переключаются в настройках

> Поддерживаемые бэкенды: **Ollama / OpenAI-совместимый** (полностью локально) · **Claude Agent** (облако Anthropic)

---

## Что умеет плагин

AI Wiki читает ваши заметки и ведёт структурированную базу знаний (вики) рядом с ними. Представьте ассистента, который читает всё, что вы пишете, и автоматически поддерживает актуальный справочник.

| Функция | Что делает |
|---|---|
| **Ingest** | Читает открытую заметку, извлекает ключевые темы (люди, инструменты, процессы, термины), создаёт страницы или обновляет существующие защищёнными патчами секций. Большой Markdown обрабатывается ограниченными чанками с полным покрытием evidence. Теги стандартизированы: страницы переиспользуют существующий словарь тегов домена, несут тег своего типа сущности, а число тематических категорий тегов ограничено настройкой домена |
| **Query** | Отвечает на вопрос, используя вики как источник; ответ появляется в панели с перекрёстными ссылками |
| **Lint** | Проверяет страницы вики на пробелы, устаревший контент и битые ссылки; выдаёт отчёт в панели |
| **Fix** | После Lint — введите инструкцию в чат панели, и ИИ применит правки |
| **Init** | Создаёт новую область знаний (домен): структуру папок, `metadata.jsonl`, `index.jsonl` и `log.jsonl` |
| **Re-init** | Удаляет и заново создаёт всё дерево домена, включая метаданные и пустые папки, затем пересобирает его из источников |
| **Format** | Улучшает форматирование любой открытой заметки (вне вики): заголовки, таблицы, frontmatter, подписи к изображениям. Показывает предпросмотр перед применением. Инвариант: факты не добавляются и не удаляются — только улучшается ясность изложения. Если заметка принадлежит настроенному домену, теги переиспользуются из существующего словаря тегов этого домена |
| **Chat** | Интерактивный диалог в боковой панели после Query или Lint для уточнения результатов |
| **Export OKF** | Сериализует домен в бандл Google [Open Knowledge Format](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/) — папку markdown с OKF-frontmatter, сгенерированными `index.md`/`log.md` и обычными ссылками `[текст](ссылка.md)` — для обмена с внешними AI-агентами и инструментами. Только на компьютере |

Fix, Format и Chat запускаются из кнопок в боковой панели — не через палитру команд. Export OKF доступен и как кнопка в панели, и как команда палитры.

> **Мобильные устройства:** работает только Query. Ingest, Lint, Init, Format и Export OKF — только на компьютере.

> **OKF-frontmatter:** страницы вики используют Google Open Knowledge Format — обязательный `type` (подпапка типа сущности) плюс `description`/`resource`/`timestamp`/`tags`/`status`. Граф знаний живёт в body-секциях `## Related` / `## External links` (на диске — Obsidian-`[[wikilinks]]`, переписываются в markdown-ссылки только при OKF-экспорте).

## Безопасность

### Запуск внешнего процесса

Бэкенд Claude Agent запускает внешний процесс для работы с Claude CLI:

- **Что запускается:** абсолютный путь, указанный в Настройках → «Путь к Claude Code» (например, `/home/user/iclaude.sh`). Перед каждым запуском путь проверяется: он должен быть абсолютным и не содержать traversal-последовательностей.
- **Почему это необходимо:** бэкенд Claude Agent вызывает `claude` / `iclaude.sh` как подпроцесс — без этого он работать не может.
- **Права доступа:** подпроцесс наследует права вашего системного пользователя — такие же, как при ручном запуске Claude CLI в терминале.
- **Согласие при первом запуске:** при первом использовании Claude Agent появляется диалог подтверждения — ничего не запускается без вашего согласия. Отозвать согласие можно, удалив `shellConsentGiven` из файла `local.json` плагина.

### Доступ к хранилищу

Плагин читает только папки, которые вы указали как источники для каждого домена. Полный обход хранилища не выполняется.

## Быстрый старт: Ollama (полностью локально)

Не нужны аккаунты и облачные сервисы — ИИ работает на вашем компьютере.

### 1. Установите Ollama

Скачайте с [ollama.com](https://ollama.com) и загрузите модель:

```bash
ollama pull llama3.2
```

### 2. Установите плагин

Скопируйте папку плагина в хранилище:

```bash
# симлинк для разработки
ln -s /path/to/obsidian-ai-wiki ~/.config/obsidian/Plugins/obsidian-ai-wiki
```

Или скопируйте папку вручную в `<vault>/.obsidian/plugins/obsidian-ai-wiki/`.

### 3. Включите плагин

Obsidian → Настройки → Сторонние плагины → найти «AI Wiki» → включить.

### 4. Настройте

Настройки → AI Wiki:

| Параметр | Значение |
|---|---|
| Backend | Native Agent (OpenAI-compatible) |
| Base URL | `http://localhost:11434/v1` |
| API Key | `ollama` |
| Модель | `llama3.2` |
| Temperature | `0.2` |
| Бюджет токенов ввода | `16384` |
| Бюджет токенов вывода | `4096` |
| Семантическое сжатие | `Сбалансированное` |

### 5. Создайте область знаний (домен)

Домен — это пара: «папка с источниками → папка вики». ИИ читает заметки из папки источников и создаёт страницы вики в папке вики.

Откройте боковую панель AI Wiki (иконка 🧠 на ленте или Палитра команд → «Открыть панель»), затем:

1. Нажмите кнопку **Init** в секции «Создание»
2. Введите название домена (например, `work` или `проекты`)
3. Укажите пути к папкам с заметками-источниками
4. Нажмите **Запустить** — плагин создаст структуру вики и начнёт наполнять её

### 6. Первый Ingest

1. Откройте любую заметку в Obsidian
2. В боковой панели выберите нужный домен в выпадающем списке
3. Нажмите кнопку **Ingest**
4. Следите за прогрессом в боковой панели — новые страницы вики появятся в папке домена

## Боковая панель

Боковая панель — главный интерфейс AI Wiki. Откройте её через иконку 🧠 на ленте или Палитру команд → «Открыть панель».

### Секции

**Создание** — кнопка **Init**: создать новый домен (название, папка вики, папки источников).

**Наполнение / Актуализация** — работа с существующим доменом:
- Выпадающий список доменов
- **↻** обновить список доменов
- **📁+** управление источниками (добавить или удалить папки источников)
- **♻** полный Re-init: удалить и заново создать всё дерево домена, затем пересобрать из источников
- 📜 открыть лог-файл домена
- 🗒 открыть индексный файл домена
- **Ingest** — обработать текущую открытую заметку
- **Lint** — проверить качество вики
- **Format** — улучшить форматирование текущей заметки

**Запрос** — введите вопрос и нажмите **Спросить**. Ответ появится в панели со ссылками на страницы вики. Используйте чат ниже для уточнений.

### Прогресс модели и Re-init

Каждый запрос к модели проходит один понятный lifecycle в панели: **Подготавливаем запрос →
Запрос передан модели → Ожидаем ответ модели → Модель формирует ответ → Проверяем ответ →
Применяем результат → Готово** (либо завершается повтором, ошибкой или отменой).
Рассуждение остаётся в раскрываемом блоке. Call site, transport, номер попытки, бюджеты и
данные провайдера сохраняются только в `agent.jsonl`, но не в подписях панели. Таймер
ожидания показывает прошедшее время в UI: это не heartbeat провайдера, и idle deadline он
не продлевает. В agent log рассуждение сохраняется упорядоченными ограниченными записями
до 4 MiB на операцию; лишний текст заменяется marker обрезки только с метаданными.

Фоновые structured-вызовы — bootstrap Init, evidence map/reduce, synthesis Ingest и
ограниченные batch-вызовы Lint — получают атомарный non-stream ответ. Интерактивные Chat,
ответ Query и Format используют SSE, поэтому рассуждение или текст ответа видны по мере
поступления.

Полный Re-init до изменений проверяет bootstrap и снимки источников, затем ровно один раз
удаляет всё дерево `!Wiki/<domain>`: страницы, метаданные, индексы, логи, временные данные,
вложенные папки типов и устаревшие пустые каталоги. Перед ingest создаются свежие metadata
и index. Ошибка удаления или конфликт параллельной записи останавливает ingest; транзакция
восстанавливает прежний снимок, когда это безопасно, и не перезаписывает параллельно
созданное дерево домена.

### Ограниченная обработка и storage

Эти настройки относятся к разным частям вызова: **Бюджет токенов ввода** ограничивает
подготовленный запрос, **Бюджет токенов вывода** — сгенерированный ответ, а **Бюджет
токенов размышления** отдельно разрешает рассуждение native-модели, если провайдер его
поддерживает. В Native Agent всеми тремя настройками управляет плагин; ввод Claude
ограничивает AI Wiki, а выводом Claude управляет CLI.

Бюджет ввода ограничивает весь подготовленный запрос, включая system/schema-инструкции,
а не только текст заметки. Если данные не помещаются, AI Wiki упаковывает целые единицы
контекста и применяет batching или разбиение для конкретной операции вместо скрытого
обрезания обязательного контента. После context error провайдера запрос может быть
переупакован с меньшим бюджетом. Настроенный бюджет остаётся явным: AI Wiki не определяет
context window модели автоматически.

Ingest делит большой Markdown по стабильным границам секций, абзацев, окон строк и
fenced-code блоков. Ограниченные map-вызовы создают evidence с привязкой к источнику, а
reduce-вызовы сохраняют полное покрытие до synthesis. Новые страницы создаются целиком.
Существующие получают защищённые хешами страницы/секции патчи `add`, `append` или
`replace`: нетронутые секции сохраняются, а устаревшее состояние не перезаписывается.

`index.jsonl` — структурированное хранилище: `page` records содержат retrieval-метаданные,
а `chunk` records — метаданные эмбеддингов и векторы. Сериализованные векторы и сырые
records индекса никогда не попадают в промты модели; prompt builders передают только
нужные выбранные evidence, Markdown-секции и разрешённые метаданные. Неизменившиеся
эмбеддинги чанков переиспользуются, если совпадают хеш embedding-текста, модель и
размерность.

Маленькие источники сохраняют короткий путь обработки. Большие источники, страницы,
истории, заметки или PDF могут потребовать дополнительных ограниченных вызовов модели:
это увеличивает задержку и стоимость провайдера в обмен на полную обработку в заданном
бюджете ввода. Vision Check — тоже реальный запрос провайдеру и может стоить небольшую
сумму.

| Параметр | Описание | По умолчанию |
|---|---|---|
| Глубина BFS | Query: сколько шагов по ссылкам делает ИИ при сборе контекста. `0` = только прямые совпадения | `1` |
| Порог хаба | Lint: страницы с числом исходящих ссылок выше порога помечаются как перегруженные | `20` |

## Синхронизация

Файл `local.json` (внутри папки плагина) хранит настройки, специфичные для вашего компьютера: путь к Claude CLI, API-ключ, выбранный бэкенд. **Исключите `local.json` из синхронизации** при использовании Obsidian Sync / git / Syncthing — иначе настройки будут перезаписаны на других машинах.

Файлы домена (`metadata.jsonl`, `index.jsonl`, `log.jsonl` внутри `!Wiki/<domain>/`) находятся внутри хранилища и синхронизируются вместе с заметками в обычном режиме. `local.json` остаётся локальным и не должен синхронизироваться.

## Ориентиры по производительности

Реальные замеры на домашнем сервере с моделью **`deepseek-v4-flash:cloud`** через Native Agent (OpenAI-совместимый endpoint). Скорость генерации — около 100–130 токенов/с, что соответствует средней локальной или самостоятельно хостящейся GPU.

| Операция | Типичная длительность | Вызовов LLM | Вх. токенов (avg/вызов) | Вых. токенов (avg/вызов) | Скорость (tok/s) |
|---|---|---|---|---|---|
| **Query** | 4–14 с | 1 | ~6 800 | ~470 | ~100 |
| **Ingest** (1 заметка) | ~25 с | 2 | ~4 300 | ~1 200 | ~109 |
| **Init** (6–24 файла-источника) | 6–27 мин | 20–35 на сессию | ~5 700 | ~2 600 | ~122 |
| **Lint** (большой домен) | ~60 мин | 69 | ~14 300 | ~6 500 | ~128 |

> **Init** и **Lint** масштабируются по числу файлов и страниц вики. Домен из 6 файлов — ~6 мин; из 24 файлов — ~27 мин. Lint большого домена (69 вызовов LLM) занял около 60 мин.

### Что влияет на скорость

- **Модель** — меньшие/квантованные модели быстрее; более крупные дают лучшее качество вики
- **Сервер вывода** — локальная GPU быстрее всего; облачные API добавляют сетевую задержку
- **Размер домена** — время Init и Lint растёт линейно с числом файлов
- **Большие входные данные** — ограниченные map/reduce, batching и сегментация добавляют вызовы ради полного покрытия
