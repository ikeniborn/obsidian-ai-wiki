Проблема

Инкрементальный реинит вечно перепомечает неизменённые источники. Детектор сравнивает mtime источника с mtime вики-страниц: (а) mtime источника поднимается на каждом ингесте; (б) за прогон переписывается лишь подмножество страниц источника, поэтому min(mtime страниц) остаётся старым → source.mtime > min(pages) всегда TRUE. Подтверждено логами _agent.jsonl.

Требования

IDСекцияТребование
R1Decision / Component 1Детект по хэшу тела источника (frontmatter отрезан), не по mtime. FNV-1a, значение fnv1a:<hex>. Чистый модуль без зависимостей.
R2Decision / Data modelХэш хранится на стороне вики в _config/_domain.json; файлы-источники для bookkeeping не пишутся никогда.
R3Data model / Migrationanalyzed_sources становится картой «путь → хэш» (единый источник правды). Миграция v3: список→карта, значения "" до baseline; флаг analyzed_sources_v3.
R4Component 2computeChangedSources переписан: входы mtime/wikiPages убраны. Правило: нет ключа→changed; ""→silent baseline; хэш≠сохранённый→changed; иначе skip. Возвращает {changed, baselined}.
R5Component 3controller.computeIncrementalPlan читает тела источников, считает хэши, передаёт карту, персистит baselined. Блок wikiPages удалён.
R6Component 4Хэш пишется в точках bookkeeping (init.ts:350-361, 438-444) — покрывает все пути ингеста (full reinit, --sources, --incremental, ручной).
R7Component 5Ретипизация analyzed_sources string[]→map в init.ts, delete.ts, types.ts. Чистка осиротевших mtime/WikiPageInfo.
R8RemovedA2-reorder больше не несущий; оставлен на месте (удаление вне scope).
R9Verificationout-of-vault unit, migration unit (идемпотентность v3), fs-интеграция, ручной e2e на ai-agent.

↓ hashSource()
хэш fnv1a:<hex>
↓

changed → ReinitModeModal → ингест выбранных → analyzed_sources[path]=hashSource(body)
