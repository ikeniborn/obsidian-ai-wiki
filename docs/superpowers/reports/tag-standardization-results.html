СекцияТребование
ProblemIngest: «reuse tags from existing pages» опирается только на похожие страницы в контексте — словаря домена LLM не видит, теги плодятся произвольно. Format: инструкция про reuse мёртвая — в контекст не подаётся ничего из vault. Единственная логика — TAG_RE, который молча выбрасывает почти-валидные теги.
Decision SummaryДинамический реестр (без ручной таксономии); скоуп — домен (wiki-страницы + источники из source_paths); в промпт идёт весь реестр без капа; модель «сущности + темы»; лимит тематических категорий (default 12, max_tag_categories); мягкий контроль (warning, тег сохраняется); детерминированная синхронизация сущностного тега; нормализация вместо отбрасывания.
Part 1 — Tag registrysrc/utils/tag-registry.ts (новый): collectDomainTags — скан frontmatter tags: wiki-папки + source_paths, нормализация + TAG_RE, агрегация по категориям с частотами; renderTagRegistryBlock — блок EXISTING DOMAIN TAGS: entity-категории + тематические (N/12), при исчерпании лимита — «reuse only». Без усечения.
Part 2 — IngestСбор реестра после определения домена; блок в user-сообщение buildIngestMessages; правка prompts/ingest.md:21 (порядок: сущностный тег → reuse из блока → новый при свободном лимите). Пост-обработка: авто-добавление тега типа сущности (по wiki_subfolder); превышение лимита категорий → warning, тег сохраняется.
Part 3 — Formatagent-runner.ts: домен из formatDomain (стр. 137) + fallback detectDomain по пути; передача в runFormat; тот же блок реестра в userInitial; правка templates/_format_schema.md:7. Домен не определился → блока нет, поведение как сейчас.
Part 4 — НормализацияnormalizeTag(): trim → срез # → lowercase → пробелы→- → \→/; перед TAG_RE в list-tags для SOURCE_RULES и WIKI_PAGE_RULES; не прошёл после нормализации — отброс с warning, как сейчас.
Part 5 — КонфигурацияDomainEntry.max_tag_categories?: number, отсутствует → 12; хранится в _domain.json, правится как JSON (без UI).
Out of Scopelint/init, ретегирование существующих страниц, кэш _tags.json, similarity-remapping, settings UI.

2. Схема решения

Общий поток: сбор реестра → рендер блока → две точки внедрения (ingest, format) → детерминированная пост-обработка.

Определение домена
ingest: detectDomain · format: formatDomain / detectDomain fallback
→ wiki_folder, source_paths, entity_types, max_tag_categories
▼
collectDomainTags
src/utils/tag-registry.ts
скан frontmatter tags: wiki-папки + source_paths; normalizeTag + TAG_RE; агрегация по категориям с частотами
renderTagRegistryBlock
entity categories (entity_types) + thematic (N/limit)
блок EXISTING DOMAIN TAGS, весь реестр, без усечения
▼
Ingest LLM-контекст
buildIngestMessages + prompts/ingest.md
порядок: сущностный тег → reuse → новый при свободном лимите
Format LLM-контекст
userInitial + templates/_format_schema.md
только тематическое reuse (источник — не сущность); нет домена → без блока
▼
Валидация frontmatter
raw-frontmatter.ts · list-tags
normalizeTag перед TAG_RE; не прошёл — отброс с warning
Ingest пост-обработка
после frontmatter repair
авто-тег типа сущности (wiki_subfolder); лимит категорий превышен → warning, тег сохранён
